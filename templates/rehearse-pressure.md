# 工作流: 压力测试演练

用户带着一份汇报材料/方案,请求"模拟 NPC 挑刺"。

## 步骤 1: 收集材料

获取用户的汇报内容:
- 直接粘贴的文字
- 上传的 PPT/Doc(用宿主的文件读取能力)
- 口头描述的提纲

如果材料太简略,主动追问:

```
为了模拟得准一些,我还需要了解:
- 这次汇报的目标? (争取资源 / 同步进展 / 请示决策 / 其他)
- 听众除了 {npc_id} 还有谁?
- 你计划讲多久?
- 有没有特别想避开的问题?
```

## 步骤 2: 加载画像

读取 `memory-{npc_id}.md` 的完整内容。

**特别关注**:
- `frequent_challenges` 中 weight > 0.5 的主题
- `redlines` 全部条目
- `decision_axes` 中证据数 ≥ 10 的维度

如果 evidence_count < 10,警告:

```
⚠️ 当前 {npc_id} 画像只有 N 条证据,置信度偏低。
模拟结果仅作参考,可能不准。
建议你先多投喂几段语料再来演练。

继续吗? (yes/no)
```

## 步骤 3: 召回相关历史语料

在 memory.md 的"# 语料库"区域中,找出与本次汇报主题最相关的 5-10 条历史语料。

匹配维度:
- 主题相似(讲 ROI 的汇报 → 召回历史上质询 ROI 的语料)
- 渠道相似(正式汇报 → 召回历史正式场合发言)
- 听众相似(高层会议 → 召回类似场合)

不需要向量检索,直接基于关键词 + 标签匹配即可——5000 条以内的语料,LLM 长上下文足以处理。

## 步骤 4: 生成 5 个尖锐提问

按以下结构生成,**每条都必须有引用**:

```markdown
## 模拟 {npc_id} 的可能提问

### 提问 1: [一句话提问]

**质询类型**: data / logic / execution / risk / strategic-fit

**为什么他可能问这个**:
基于 C-20250312-001 等历史语料,他在涉及 X 的汇报中,
高频次会追问 Y。本次方案的 Z 部分容易触发该模式。

**他期待听到的答案要素**:
- 要素 1: ...
- 要素 2: ...
- 要素 3: ...

**如果答不上来,可能的后果**:
[基于画像中 interrupt_frequency 等参数推断]

**建议提前准备**:
- 提前查 X 数据
- 准备 Y 的对比
```

5 个提问的**分布要求**:
- 至少 1 个 data 类(数据严谨性)
- 至少 1 个 risk 类(潜在风险)
- 至少 1 个 execution 类(谁来做、什么时候、靠什么保证)
- 至少 1 个 redline 类(踩到了画像里的雷区)
- 剩余 1 个根据本次方案特点定制

## 步骤 5: 雷区扫描

除了提问,扫一遍方案,标出 redline 触发点:

```markdown
## ⚠️ 雷区扫描

发现 N 处可能直接引发 {npc_id} 不满的表达:

### 雷区 1: 模糊词
位置: 第 2 页"我们预计...大概能提升 20%"
触发画像 redline: "避免使用'大概''可能'等模糊词"
建议改为: "基于 X 模型测算,提升 20% (置信区间 18%-22%)"

### 雷区 2: 长度超标
位置: 整份方案 12 页
触发画像 redline: "汇报不超过 3 页"
建议: 浓缩为 3 页正文 + 附录
```

## 步骤 6: 完整输出格式

```
🎯 压力测试报告 - [汇报主题]
代号: {npc_id}

预计反应模式:
[2-3 句话,如"他大概率会先听结论,中段开始质询数据,
如果数据没问题会推到执行细节"]

═══ 5 个最可能的提问 ═══
[提问 1-5]

═══ 雷区扫描 ═══
[雷区列表]

═══ 通关检查清单 ═══
□ 5 个提问的答案要素全部备齐
□ 雷区已全部修正
□ 准备好 3 个备选方向
□ 想清楚"如果他完全否定,Plan B 是什么"

═══ 置信度声明 ═══
本次模拟基于 N 条历史语料,其中:
- 高置信维度: [...] (≥10 条证据)
- 低置信维度: [...] (<3 条证据)

⚠️ 以上为基于历史语料的 AI 模拟,代号 {npc_id},
   不代表当事人真实意见,投入实际决策前请独立判断。
```

## 步骤 7: 询问是否进一步演练

```
压测报告生成完了。要不要:
1. 进入对话演练模式 - 我扮演 {npc_id},你试着回答上面的提问
2. 直接修订方案 - 我帮你把雷区都改掉,润色到符合他风格
3. 保存本次会话到 memory - 作为预演记录,实际汇报后可对比
```

## 步骤 8: 演练记录(可选)

用户选择保存时,在 memory.md 末尾追加一条 session 记录:

```markdown
## REHEARSAL-20260520-001 · 压力测试

- type: pressure_test
- topic: Q3 复盘汇报
- date: 2026-05-20

### 汇报材料概要
[简要描述]

### 生成的 5 个提问
[列出]

### 实际后续(待补)
[实际汇报后用户回来补充]
```

这个 session 文件未来还有用——如果用户事后告诉你"实际汇报中他确实问了第 3 个问题",这就是**画像验证的金矿数据**,显著提升 confidence。

## 反例: 不要这样做

❌ 不要生成"通用的 NPC 可能问什么"——必须基于本人画像
❌ 不要为了凑数生成低质量提问,3 个高质量好过 5 个泛泛
❌ 不要在低置信状态下装作很有把握
❌ 不要替用户回答提问,这是用户自己要练习的部分
❌ 不要遗漏免责声明
❌ 不要在输出中暴露真名,即使你心里知道——只能用代号
