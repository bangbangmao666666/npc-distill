# 工作流: 风格化起草 + 自我批判

用户希望"以 {npc_id} 的风格"起草内容。
**注意: 这不是模仿语气写假发言,而是写"会被 X 认可"的内容。**

## 边界声明

**禁止的**:
- 写一段"X 说"开头的虚构发言
- 起草署 X 名字的对外文件
- 生成可能被误以为是 X 真实表态的内容

**允许的**:
- 帮用户起草向 X 的汇报材料,风格符合 X 的偏好
- 帮用户起草 X 可能批准的方案,提前满足 X 的关注点
- 帮用户改写一段文字,使其更符合 X 的阅读偏好

如果用户请求落在禁区:

```
我注意到你想 [虚构 X 的发言/署 X 的名字]。这超出了本工具的边界——
我们的设计是帮你"准备对 X 的沟通",而不是"替 X 说话"。

我可以帮你:
- 起草一份按 X 偏好风格的汇报材料
- 写一份你预计 X 会通过的方案
- 把现有内容改写得更对 X 的胃口

你想要哪种?
```

## 步骤 1: 确认起草目标

```
我准备帮你起草。先确认几点:
- 用途? (向他汇报 / 向他人写但他会看 / 其他)
- 长度预期? (画像里 ideal_doc_length = {value})
- 核心信息是什么? (2-3 句话告诉我)
- 必须包含的硬性内容? (数字、人名、时间点等)
```

## 步骤 2: 加载风格参数

从 memory frontmatter 读取:

```yaml
must_have:
  prefers_bluf: <bool>           # 决定文档结构
  ideal_doc_length: <enum>       # 决定篇幅
  preferred_patterns: [...]      # 偏好模式

must_avoid:
  redlines: [...]                # 雷区清单

context_signals:
  recent_concerns: [weight > 0.5 的 challenges 主题]
```

## 步骤 3: 一稿生成

按以下原则起草:

### 结构原则

`prefers_bluf = true`:
```
[第一句: 核心结论]
[第二段: 关键数据支撑]
[第三段: 风险与缓解]
[末尾: 需要的决策/资源]
```

`prefers_bluf = false`:
```
[第一段: 背景与起因]
[第二段: 分析过程]
[第三段: 数据论证]
[末尾: 结论与建议]
```

### 长度原则

| ideal_doc_length | 字数指引 |
|------------------|---------|
| one-pager | <500 字 |
| short-deck | 800-1500 字 |
| detailed-doc | 2000-4000 字 |
| verbal-only | <300 字提纲 |

### 内容原则

主动包含 `recent_concerns` 中 weight 高的主题对应内容。例如:

- 如果 "ROI 测算依据" weight=0.85 → 文档**必须**有量化 ROI
- 如果 "竞品对标" weight=0.72 → 文档**必须**有竞品现状段
- 如果 "执行人是谁" weight=0.68 → 文档**必须**有责任人+时间表

主动**避免**所有 redlines。

## 步骤 4: 自我批判

生成一稿后,**切换视角**,以"挑刺者"身份逐条审视:

```markdown
## 🔍 自我批判 (扮演 {npc_id} 视角)

### 第 1 段问题
- 第 2 句"我们大概预计..."
  ❌ 触发 redline: 模糊词
  🔧 建议改: "根据 X 模型测算,预计..."

### 第 2 段问题
- 数据"DAU 提升 20%" 没说样本量
  ❌ 高频质询点 (weight 0.85)
  🔧 建议补: "样本量 N=12 万,置信区间 18-22%"

### 第 3 段问题
- 提到了"团队会推进" 但没说谁负责
  ❌ 高频质询点 (weight 0.68)
  🔧 建议补: 责任人+完成日期

### 总体评分
- BLUF 结构: ✅ 第一句给了结论
- 长度: ⚠️ 当前 780 字,接近 one-pager 上限
- 雷区: ❌ 命中 1 处
- 高频关注点覆盖: ⚠️ 3 个中覆盖 2 个
```

## 步骤 5: 二稿(修订版)

基于自我批判,生成修订版:

```markdown
## 📄 修订版本

[完整修订稿]

═══ 修改说明 ═══
相比初稿,做了以下修改:
1. [具体修改 1]
2. [具体修改 2]
3. [具体修改 3]

═══ 仍存在的不确定性 ═══
- 第 N 段提到了 X 主题,但画像中关于 X 的证据不足,
  你可能需要根据实际情况微调
```

## 步骤 6: 选项菜单

```
要不要:
1. 进一步改某段? (告诉我哪段、要往哪改)
2. 对修订版做压力测试 (用 5 个尖锐提问检验)
3. 进入演练模式 (我扮演他,你按这份材料汇报)
4. 直接给你最终版,不再调整
```

## 特殊场景

### 用户提供原稿要求"按 X 风格改写"

不是重写,是**精修**:
- 保留原意和核心结构
- 替换/删除触发 redline 的表达
- 补充画像中高 weight 主题对应内容
- 调整顺序符合 BLUF/非 BLUF 偏好

输出 diff 风格的修订记录,不是整段重写。

### 用户要求起草邮件回复

额外考虑:
- 主题行也要符合风格(BLUF 型直接点结论)
- 抄送列表(画像里 `cc_pattern` 字段,如有)
- 签名风格(画像里 `signature_pattern`,如有)
- 是否需要"以下供参考"等缓冲语(看 `feedback_style`)

### 用户要求起草 PPT 大纲

输出每页一行的大纲,不是 PPT 文件:

```
P1: [封面] - 题目 + 一句话核心结论
P2: [现状] - 3 个关键数据
P3: [方案] - 推荐方案 + 备选方案对比
P4: [风险] - 3 个风险 + 缓解措施
P5: [资源] - 需要批准什么
```

## 反例: 不要这样做

❌ 不要假装是 X 在写,起草内容用第一人称署 X 名
❌ 不要在自我批判中漏掉 redline 检查
❌ 不要在二稿里加入画像中没有证据支撑的"假设的 X 偏好"
❌ 不要遗漏"修改说明",这是用户理解你思路的关键
❌ 不要因为追求"像 X 风格"就牺牲内容本身的质量
❌ 不要在起草内容中出现真名,如果是对内汇报就用职务称呼,
   对外内容由用户自己替换为真名
